Built Different, By Design

Every component of FERROS was designed from scratch with safety, sovereignty, and human consent as non-negotiable first principles.

🦀

Rust-Native Kernel

Zero unsafe code in the critical path. The entire microkernel is written in safe Rust with compile-time memory guarantees, eliminating entire classes of CVEs before the CPU even boots.

no_std + alloc
🔐

Consent Engine

Fine-grained capability tokens govern every inter-process communication. Users and local processes grant, revoke, and audit permissions in real-time — no ambient authority anywhere in the system.

capability tokens
🌐

HTML Renderer

A lightweight, sandboxed HTML/CSS renderer runs directly in userspace, enabling rich local UIs without a browser engine. No V8, no Electron — just pixels from your own hardware.

sandboxed userspace
🛡️

Capability-Based Security

Every resource — files, sockets, devices — is accessed through an unforgeable capability handle. Object-capability model enforced by the kernel ABI; no privilege escalation path exists.

object-capability model
🔄

Local-First Sync

CRDTs and append-only logs power peer-to-peer state synchronisation without a central server. Your data lives on your hardware and replicates on your terms, over any transport.

CRDT · append-only
🤖

Agent Hosting

Run autonomous AI agents in isolated capability-sandboxed processes. Each agent declares its required capabilities at launch; the Consent Engine approves, limits, and monitors in real time.

isolated agent sandbox

For Contributors

FERROS is a community project. Every discipline is needed — OS engineers, Rust developers, hardware hackers, UX designers, and technical writers.

1

Read the Blueprint

Start with ferros-blueprint.html — it defines the system philosophy, component boundaries, and open questions. Understanding the why before the how saves everyone time.

2

Set Up the Toolchain

Install Rust nightly with rustup default nightly, add the x86_64-unknown-none target, and install QEMU. Run make check to verify the environment.

3

Pick a Module

Check the issue tracker for good-first-issue tags. Each module has a DESIGN.md describing its interface contracts and what help is currently needed.

4

Open a Draft PR Early

Don't wait for perfection. Open a draft PR as soon as you have a working skeleton. The team reviews architecture before implementation to avoid wasted effort.

5

Join the Community

Real-time discussion happens in the project Matrix room. Design RFCs are published as GitHub Discussions and remain open for comment for at least two weeks before merging.

Where FERROS Runs

Designed for the edges of the network — from the device in your pocket to the server in your home rack.

🏠

Smart-Home HUD

Run FERROS on a wall-mounted display. The HTML renderer drives a local dashboard of sensor data, schedules, and controls — no cloud account needed, no subscription, no telemetry.

Raspberry Pi 4 HDMI output touch input local-only
🔗

P2P Coordination Node

Deploy lightweight FERROS nodes that sync state over local network or Bluetooth without a central broker. CRDT logs ensure eventual consistency even when devices go offline for days.

CRDT sync mesh networking offline-first low power
🌐

Federated Nodes

Multiple FERROS instances federate via DIDs and mutual capability grants. Share compute, storage, and services across a trusted group without any centralised infrastructure or third-party identity provider.

DID identity federated auth shared VFS encrypted
🤖

AI Agent Hosting

Give AI agents a sandboxed home. Each agent gets a capability manifest, a resource budget, and a monitoring dashboard. Revoke access instantly if behaviour deviates from policy — no kernel patch required.

Wasm sandbox resource budgets live monitoring instant revoke
