02   Architecture Decision: Linux Now, FERROS Later

🐧 Phase D-0 through D-6: Linux on All Nodes

All home lab nodes (Raspberry Pi 5, Jetson Orin Nano) run Ubuntu or Debian ARM from day one. This is practical: driver support is mature, deployment is fast, and all FERROS HTML interfaces run without modification inside any modern browser on Linux. The command node Ubuntu desktop is the primary development machine.

🦀 Parallel Track: FERROS Kernel in QEMU

FERROS kernel development happens entirely in QEMU on the dev box, in parallel with the live deployment. This keeps risky kernel work isolated. There is no requirement to boot FERROS on real hardware until the kernel is stable enough to replace Linux on a node (Phase D-7+).

🔄 Migration Path (Phase D-7+)

When the FERROS kernel reaches a stable HTML-rendering milestone (Phase 0 of the kernel roadmap), a single edge node is converted from Linux to FERROS as a proof-of-concept. Rollout to additional nodes follows as each subsystem (storage, networking, display) reaches production quality.

// Runtime matrix
Command Node → Ubuntu 24.04 LTS (dev + QEMU host)
Pi 5 Nodes → Ubuntu Server 24.04 ARM (HUD display, schedule)
Jetson Nodes → JetPack 6 / Ubuntu ARM (camera AI, inference)
FERROS Kernel → QEMU/KVM only until Phase D-7+

07   Risk Register

⚠️ Risks, Severity, & Mitigations

13 Identified Risks
Risk Category Severity Mitigation
GPU availability / cost spike Hardware High Use existing desktop GPU initially; defer AI cluster to D-5. Plan for RTX 4090 alternative (A6000 refurb).
botgen-rust API bridge complexity Timeline High Scope D-2 to a minimal read-only status feed first. Bidirectional control can follow in D-3.
Jetson JetPack driver friction Hardware Medium Follow NVIDIA JetPack 6 official docs. Budget 1–2 weeks for driver setup. Use pre-tested Ubuntu image.
Home network / ISP variability across 7 homes Network Medium WireGuard VPN mesh tolerates varied ISPs. Use dynamic DNS. Graceful offline mode for all HTML interfaces.
Physical deployment coordination (6 remote homes) Logistics Medium Pre-configure hardware before shipping. Remote SSH + WireGuard for post-deployment support. Ship to 1 beta home first.
Command node as single point of failure Resilience High UPS backup for command node. Edge nodes operate in local-only mode if command node is offline. Implement health checks.
SD card corruption on Pi nodes Hardware Medium Use read-only SD card mount for HUD. Maintain NFS-boot or USB-SSD fallback. Keep reflash script.
FERROS kernel timeline slippage Timeline Low (isolated) Kernel work is parallel track. D-0 through D-6 are fully independent of kernel status. No deployment blockers.
Privacy / consent at other homes Governance Critical Explicit written consent from each home occupant before installation. Camera feeds stay local per home. No cross-home video access without explicit approval.
AI model inference costs / VRAM limits Technical Medium Start with quantized 7B–13B models. 24 GB VRAM (RTX 4090) supports most practical models. Edge offload reduces command node load.
localStorage data loss on browser clear Technical Low Export/import JSON backup feature in schedule-ledger and personal-profile pages. Warn user before clearing.
Scope creep delaying D-0 through D-2 Timeline High Strict phase gating: each phase must reach its success criteria before beginning the next. Kernel work never blocks deployment.
Hardware supply chain delays Hardware Medium Order Jetson Orin Nano and Pi 5 units early (lead time 2–4 weeks). Use Pi 4 as fallback for HUD display.
