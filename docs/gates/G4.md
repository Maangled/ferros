# Gate G4 — Launch

**Status:** 🟡 Active  
**Stream:** S7  
**Unlocks:** coordinated lab rollout and controlled test-home preparation; `v0.2.0` tag

---

## Entry criteria

- G3 is closed (agent center on runtime demo; `ferros-node demo` green in CI).
- S7 implementation work is underway.
- At least one home-server or Raspberry Pi is available for testing.

---

## Required evidence

All of the following must be true simultaneously. Evidence must be documented in this file under the Evidence section before G4 is declared closed.

- [ ] `ferros-hub` binary cross-compiles for `aarch64-unknown-linux-gnu` (or `x86_64-unknown-linux-gnu` for home server).
- [ ] `ferros-hub` runs on a physical device (not CI, not QEMU, not developer laptop).
- [ ] Device profile created via `ferros profile init` on the target device; persists after restart.
- [ ] `ferros agent list` shows at least one registered agent on the running hub.
- [ ] Consent enforced: a capability request that was not granted is denied at the bus level, logged, and the denial is visible on an operator-visible FERROS surface such as `ferros agent logs`.
- [ ] Hub survives a full power cycle: after reboot, profile loads, agents re-register, and core runtime state is restored without manual intervention.
- [ ] At least one coordinated clean install or reprovision confirmed on a second lab-controlled device or fresh target image.
- [ ] `docs/hub/reference-hardware.md` documents the tested hardware.
- [ ] `LAUNCH.md` is updated with the confirmed install date and hardware specification.

Optional integration lanes such as Home Assistant, local LLM runtimes, and external LLM APIs may advance in parallel, but they do not block G4 closure unless a later launch revision explicitly promotes one of them back into the core gate.

---

## Exit criteria

All required evidence items checked and documented in this file. `v0.2.0` tagged. Coordinated rollout can expand from the primary server to additional lab devices and then into controlled test homes. No unmanaged public-install ask until that controlled phase produces stable results.

## Optional Module Lanes

Home Assistant remains a useful integration lane, but it is not a current G4 blocker. Existing Home Assistant findings packets, including [docs/hardware/findings/FINDINGS-pack-c-session-03-agent-center-entity.md](../hardware/findings/FINDINGS-pack-c-session-03-agent-center-entity.md), [docs/hardware/findings/FINDINGS-pack-c-session-05-ha-restoration.md](../hardware/findings/FINDINGS-pack-c-session-05-ha-restoration.md), and [docs/hardware/findings/FINDINGS-pack-c-session-06-windows-fresh-host.md](../hardware/findings/FINDINGS-pack-c-session-06-windows-fresh-host.md), remain valid module-lane evidence and a record of what the current bridge path does or does not restore automatically.

Strict unmanaged independent installs are deferred to the later controlled test-home rollout. The current G4 install bar is one coordinated clean install or reprovision on additional lab-controlled hardware.

Local LLM runtimes and external LLM APIs are valid candidate module lanes alongside or ahead of Home Assistant.

## Blocking streams / issues

G4 is the launch gate. After G4:

- Tag `v0.2.0`.
- Continue coordinated rollout across additional lab devices.
- Begin the controlled test-home phase before asking for unmanaged installs.
- Address critical issues before any wider announcement.
- Post-launch scope (v0.3.0+) planning begins.
