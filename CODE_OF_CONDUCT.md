# FERROS Code of Conduct

FERROS is built by people working across architecture, runtime, UX, documentation, and launch-critical system design. The project benefits from direct communication, but directness is not a license for disrespect.

This code of conduct defines the minimum standard for participating in the repo, planning surfaces, and related project spaces.

---

## Expected Behavior

Participants are expected to:

- be honest about what is implemented, tested, or still unknown
- respect other contributors' time by staying scoped and concrete
- criticize ideas, claims, and implementations without attacking people
- document meaningful decisions instead of relying on hidden context or status games
- accept correction when evidence shows a claim or change is wrong
- prioritize user safety, consent, and sovereignty over hype or convenience

---

## Unacceptable Behavior

The following behavior is not acceptable in FERROS spaces:

- harassment, threats, intimidation, or sustained hostility
- discriminatory or demeaning language based on identity or protected characteristics
- deliberate misrepresentation of project status, evidence, or authorship
- spamming, trolling, or repeatedly derailing technical work
- publishing private or security-sensitive information without permission
- retaliating against someone for reporting a conduct or security concern

---

## Scope

This code applies to:

- repository discussions, issues, PRs, and reviews
- stream planning and governance documents in the repo
- direct project communication where a participant is acting in a FERROS capacity

---

## Reporting

If someone violates this code of conduct:

1. Contact the maintainers privately using the ownership information in `CODEOWNERS`.
2. Include links or concrete context so the report can be assessed fairly.
3. Do not use the public issue tracker for personal conduct reports unless there is no safer option.

For vulnerability disclosure, use [SECURITY.md](./SECURITY.md) instead of this file.

---

## Maintainer Responsibilities

Maintainers are expected to:

- enforce this code consistently and in good faith
- distinguish technical disagreement from abusive behavior
- protect reporters from retaliation where possible
- ask for corrections, pauses, or removal from project spaces when behavior crosses the line

Responses may include:

- a request to stop or change the behavior
- moderation of a discussion thread
- temporary loss of participation privileges
- removal from project spaces in serious or repeated cases

---

## Project Standard

FERROS favors rigor over theater. Be direct. Be accurate. Do not hide uncertainty. Do not turn disagreement into contempt.
