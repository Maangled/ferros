Core Philosophy

#philosophy
01
Memory Safety as Infrastructure
Every component is written in Rust unless a hardware interface absolutely requires a minimal, audited unsafe boundary. Unsafe code is quarantined, documented, and tracked in a public registry.
02
Reduced Kernel Attack Surface
Microkernel or hybrid-micro architecture. Drivers, filesystem handlers, and network stacks run in isolated user-space processes. The kernel handles only memory management, scheduling, and IPC.
03
Reproducible Builds
Every official build must be reproducible from source. Build environments are containerized and version-pinned. Binary artifacts are hashed and signed. Anyone can independently verify.
04
Observable, Accountable, Reviewable
Every major subsystem exposes structured health telemetry, version attestations, and dependency graphs. Configuration changes are logged with tamper-evident records.
05
Secure by Default
Services start with no permissions and acquire them through explicit grants. No process has more access than its declared purpose requires.
06
Privacy by Design
Personal data is encrypted at rest and never exposed without user-initiated consent. There are no silent data flows.
07
User-Readable Consent
Every permission request is one sentence, in plain language, with clear purpose, duration, and scope. Unreadable terms and conditions are programmatically banned.
08
Local-First Sovereignty
All user data lives on the local device first. Cloud services and remote coordination are optional layers, never requirements for core functionality.
09
Distributed by Design
When coordination is needed, the system uses end-to-end encrypted channels, CRDTs for eventual consistency, and capability-scoped trust delegation. No central server is ever required.
10
Agent-Ready Architecture
Designed from the ground up to host autonomous AI agents that operate under explicit human governance, with every agent action logged, consent-gated, and revocable.

System Architecture

#architecture
🔑 Bootloader & Early Boot
Rust-native UEFI application. Verifies kernel image signature against a local trust anchor, initializes framebuffer for early visual feedback, sets up initial page tables, supports measured boot with local attestation log.
UEFI Rust-native Measured Boot
⚙️ Kernel Strategy
Microkernel-hybrid design. Provides virtual memory management with hardware-enforced process isolation, pre-emptive scheduling with priority-based and fair-share policies, and typed capability channels for IPC.
Microkernel Capability IPC Pre-emptive
💾 Process & Memory Model
Each process runs in its own address space. Slab allocator for kernel objects, buddy allocator for user-space pages. Shared memory requires explicit capability grants. Memory ownership tracked at compile time and runtime.
Isolated Slab + Buddy
🔌 Driver Model
Drivers are user-space processes receiving hardware access through MMIO capability grants. Driver registry manages discovery, loading, versioning, and hot-replacement. Faulty drivers are restarted without kernel involvement.
User-space AHCI · NVMe · virtio
📁 Storage Subsystem
Log-structured, copy-on-write filesystem with cryptographic checksums at every block. File-level encryption with user-derived keys. Atomic snapshots for upgrades and rollback. Phase 0 requires read-only access only.
CoW FS Encrypted Snapshotable
🌐 Networking Subsystem
User-space TCP/IP stack in Rust, inspired by smoltcp, extended with TLS 1.3 as a first-class primitive. All connections encrypted by default. Supports DNS-over-HTTPS and DNS-over-TLS. Required Phase 3+.
TLS 1.3 smoltcp-inspired Phase 3+
🖥️ Display & UI Subsystem
Framebuffer from UEFI or GPU driver. Rust-native 2D compositor with alpha blending, damage tracking, and surface layering. Font rendering via fontdue or ab_glyph with bundled font files — no external font services.
Compositor fontdue · ab_glyph
📄 HTML Rendering Path
Rust-native HTML5 parser + CSS layout engine. Handles semantic HTML5, embedded CSS, full box model, flexbox, CSS grid, color values, gradients, border-radius, and basic keyframe animations. No JavaScript, no external resources for Phase 0.
HTML5 CSS Grid Flexbox No JS
📦 Package & Update Model
Content-addressed, signed archives. Each package declares capabilities, dependencies, and required permissions in a machine-readable manifest. Atomic updates staged and swapped. Rollback always available.
Atomic Signed Rollback
🔐 Identity & Permission Layer
Local-first identity. Cryptographic keypairs stored in hardware-backed secure storage (TPM, secure enclave). Permissions are capability tokens: unforgeable, scoped, delegatable, and revocable at any time.
Local-first DIDs TPM · Enclave
📜 Smart Contract Integration
Sandboxed contract runtime bridge. No direct kernel access. Communicates exclusively through the consent engine. Abstracts chain-specific details behind a common API. Supports multiple ledger backends.
Sandboxed Multi-chain
🔄 Distributed Sync
CRDTs for eventual consistency without central coordination. Sync channels end-to-end encrypted. Direct P2P sync on local network. Relay-assisted sync through untrusted intermediaries that cannot read relayed data.
CRDTs E2E Encrypted P2P

Security Model

#security

FERROS assumes deployment in partially hostile environments: untrusted networks, shared physical spaces, coordination through decentralized ledgers where not all participants are known or trusted, and operation alongside autonomous agents whose behavior must be constrained.

Asset Traceability
Every deployable component has a content-addressed identity and provenance record. Local software bill of materials maintained.
Integrity Verification
At every boot, critical components are verified against signed manifests. Runtime integrity monitoring detects unexpected modifications.
Signed Modules
All loadable modules must carry a valid signature from a key in the system's trust anchor set. Attestation reports for remote verification.
Encrypted Personal Data
Personal data encrypted at rest using user-derived keys. Transparent to authorized applications. Automated key rotation.
Permission-Scoped Access
Every resource access mediated by capability tokens checked at every IPC boundary. Capabilities cannot be forged or escalated.
Tamper Evidence
System logs, configuration changes, and permission grants recorded in an append-only, hash-chained log. Exportable for external audit.
Human-Readable Authorization
Every permission request must be a single sentence in plain language. System programmatically rejects requests exceeding a character limit.
Service Compartmentalization
Services run in isolated address spaces with minimal IPC surface. A compromised service cannot access other services' resources.
Auditing & Rollback
Every system update creates a recoverable snapshot. Rollback is a single operation. Audit trails cover authorization, timing, and changes.
Secure Messaging
Inter-service and inter-device messaging uses end-to-end encrypted channels with forward secrecy. Message integrity verified at transport and application layers.
Federated AI Processing
AI inference requiring external compute uses privacy-preserving techniques. Data never leaves device in raw form. Each class of AI processing requires explicit authorization.
Measured Boot
Where hardware permits, boot measurements are stored in a local attestation log. Boot chain verified before kernel execution begins.

Privacy & Consent Model

#consent

FERROS implements consent as a kernel-adjacent service. Every data access by any entity must pass through the consent engine. Consent is revocable, traceable, minimal, time-bounded, and purpose-bound. There is no "Allow All" or "Remember Forever."

Example Consent Prompts

Allow Dr. Rivera to view your heart-rate records for 30 days.
Allow your home energy service to read thermostat data this week.
Allow this assistant to summarize your grocery budget today.
Allow the firmware updater to access your smart lock configuration for 2 hours.
Allow your partner's device to view shared calendar entries until you revoke access.
Allow the cleaning-schedule agent to read room-occupancy sensor data until Friday.
Allow this trading bot to execute limit orders up to $50 per day for the next 7 days.

Risk Register

#risks
Risk Severity Mitigation Strategy
Driver compatibility HIGH virtio-first strategy, reference hardware set, community driver bounties
Hardware support breadth HIGH 2–3 defined reference boards with published hardware compatibility list
HTML/CSS rendering fidelity HIGH Strict CSS subset, this document as conformance test, incremental expansion
Usability for non-engineers HIGH Non-technical user testing from Phase 1, one-sentence consent as a hard constraint
Upgrade safety HIGH Dual-partition boot with automatic rollback; verified recovery path before every bootloader upgrade
Agent safety HIGH Capability-constrained, consent-gated, logged, and killable agents; no implicit access
Performance / IPC overhead MEDIUM Shared-memory fast paths, continuous benchmarking from Phase 0
Governance complexity MEDIUM Small named council, published decisions, gradual decentralization
Distributed trust assumptions MEDIUM Timeouts, fallbacks, and local-first operation when external systems unavailable
Legal/compliance MEDIUM GDPR/HIPAA/PCI-DSS from architecture layer: data residency controls, audit logs, deletion capabilities
Contributor sustainability MEDIUM Phased roadmap with achievable milestones; each phase delivers a usable artifact
Overpromising scope LOW Precise scope targeting a specific deployment profile, not general-purpose computing

AI Agent Hosting & Migration Path

#agents
Migration Timeline
1
Current State
AI agents (e.g., botgen-rust) run on Discord, managed through cloud-hosted backends, with control flowing through third-party APIs.
2
Intermediate State
Agent backends migrate to self-hosted servers running conventional Linux, establishing operational independence from Discord while maintaining compatibility during transition.
3
Target State (Phase 6)
Agent backends run natively on FERROS, hosted on self-owned hardware. Control, governance, and consent flow through the FERROS consent engine.
FERROS Must Provide for Agent Migration
  • Rust-compatible agent execution environment
  • Network connectivity for agent communication
  • Consent-gated API for agent access to system resources
  • Monitoring dashboard for agent activity visible in HUD
  • Hot-deploy, update, and rollback of agent code through package manager
  • Capability-scoped sandbox with no implicit resource access
  • Tamper-evident audit log for all agent actions

Primary Deployment Targets

#targets
🏠 Secure Smart-Home HUD Kiosk
Dashboard device serving as household control surface. Device status, energy usage, automation schedules, security monitoring, and family coordination rendered through the HTML-based UI pipeline. Dashboard customization requires only HTML/CSS knowledge.
🔗 P2P Coordination Devices
Peer-to-peer exchange supporting trading, service requests, contract signing, and resource sharing between devices and users without centralized intermediaries. Trusted device orchestration with mutual attestation.
🌐 Federated Infrastructure Nodes
Network nodes participating in federated services, relay-assisted sync, and distributed compute for privacy-preserving AI inference. Always local-first; federation is additive, not required.
🪪 Identity-Sovereign Personal Computing
Platform for users who require full ownership of their digital identity, data, and compute. DID integration optional; local-only identity is a first-class citizen with no functionality penalty.
🤖 AI Agent Host Systems
Self-hosted infrastructure for autonomous AI agent workloads migrated from external platforms. Agents operate under explicit human governance with capability-scoped sandboxes, consent gating, and killable execution.
